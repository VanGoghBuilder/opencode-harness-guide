Use it to tell coding agents what is real today, what is still missing, and how careful they should be about making assumptions.

## Working rules for agents

- Do not invent commands, files, or frameworks
- Treat unknowns as `TBD`
- Prefer small, additive changes
- Match current repository reality, not future intent
- Update this file when project facts change

## Command status

Use wording like this until real tooling exists:

- `Install command: TBD once a package manager is selected.`
- `Lint command: TBD once lint tooling is added.`
- `Test command: TBD once a test framework is chosen.`
- `Single test command: TBD once the test runner is chosen.`
- `Build command: TBD once buildable artifacts exist.`

## Documentation expectations

- Say what the repository is for
- Say who it is for
- Point to the best starting file
- Mark placeholders clearly

## Safe-change rule

If the repository is early-stage, choose the smallest useful next step.
Do not turn a starter file into a long policy document unless the project actually needs one.
